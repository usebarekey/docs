## Prerequisites

- Node.js 18+ or Bun 1.0+ (for the SDK)
- A Barekey account — [sign up at barekey.dev](https://barekey.dev)

---

## 1. Install and authenticate the CLI

<Tabs>
  <Tab title="Bun">
    ```bash
    bun add -g @barekey/cli
    ```
  </Tab>
  <Tab title="npm">
    ```bash
    npm install -g @barekey/cli
    ```
  </Tab>
  <Tab title="pnpm">
    ```bash
    pnpm add -g @barekey/cli
    ```
  </Tab>
  <Tab title="yarn">
    ```bash
    yarn global add @barekey/cli
    ```
  </Tab>
</Tabs>

Run the device login flow. This opens your browser to authorize the CLI session:

```bash
barekey auth login
```

The CLI prints a short user code and a URL:

```
Open your browser and visit:
  https://barekey.dev/auth/device

Enter code: ABCD-1234

Waiting for authorization...
```

After you approve in the browser, the CLI receives an access token (valid 1 hour) and a refresh token (valid 30 days). The token is stored locally and used automatically for all subsequent commands.

Verify the session:

```bash
barekey auth whoami
# → Authenticated as: your@email.com (org: acme-42)
```

## 2. Create a `barekey.json` config file

In your project root, create a `barekey.json` file that tells the CLI and SDK which org, project, and stage to target:

```json barekey.json
{
  "org": "acme-42",
  "project": "backend-api-1234",
  "stage": "development"
}
```

| Field | Type | Description |
| ------- | ------ | ----------- |
| `org` | string | Your org slug (visible in the dashboard URL) |
| `project` | string | Project slug |
| `stage` | string | Stage name — one of `development`, `staging`, `production` or a custom stage |

You can override any of these at runtime using `--org`, `--project`, and `--stage` flags. CLI flags take precedence over `barekey.json`.

---

## 3. Create your first variable

```bash
barekey env new DATABASE_URL --type string
```

Set its value:

```bash
barekey env set DATABASE_URL "postgres://localhost:5432/myapp_dev"
```

Confirm it was created:

```bash
barekey env list
# NAME          KIND     TYPE
# DATABASE_URL  secret   string
```

Read a single value:

```bash
barekey env get DATABASE_URL
# postgres://localhost:5432/myapp_dev
```

## 4. Read variables in your application

Install the SDK:

<Tabs>
  <Tab title="Bun">
    ```bash
    bun add @barekey/sdk
    ```
  </Tab>
  <Tab title="npm">
    ```bash
    npm install @barekey/sdk
    ```
  </Tab>
  <Tab title="pnpm">
    ```bash
    pnpm add @barekey/sdk
    ```
  </Tab>
  <Tab title="yarn">
    ```bash
    yarn add @barekey/sdk
    ```
  </Tab>
</Tabs>

Create a Barekey client and fetch a variable at request time:

```typescript
import { BarekeyClient } from "@barekey/sdk";

const barekey = new BarekeyClient({
  baseUrl: "https://api.barekey.dev",
  orgSlug: "acme-42",
  projectSlug: "backend-api-1234",
  stageSlug: "development",
  auth: {
    getAccessToken: async () => process.env.BAREKEY_ACCESS_TOKEN!,
  },
});

// Fetch and coerce to the declared type
const dbUrl = await barekey.env.get("DATABASE_URL").string();
console.log(dbUrl); // postgres://localhost:5432/myapp_dev
```

For a server-side integration, the access token is a CLI token (`bk_at_...`) stored in a real environment variable, or a Clerk JWT if your backend already issues those.

---

## 5. Pull variables to a local `.env` file

For local development you can pull all variables for a stage to a `.env` file:

```bash
barekey env pull --format dotenv --out .env.local
```

This writes a file like:

```bash .env.local
DATABASE_URL="postgres://localhost:5432/myapp_dev"
STRIPE_KEY="sk_test_..."
```

Add `.env.local` to `.gitignore`. Re-run `barekey env pull` whenever variables change upstream.

<Warning>
  Pulled `.env` files contain plaintext secrets. Never commit them to version
  control.
</Warning>
