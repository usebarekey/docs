Barekey stores and serves encrypted environment variables for developer teams. Variables are scoped to an **org → project → stage** hierarchy, encrypted at rest with per-project keys, and served at runtime to your application — either through the SDK or directly via the HTTP API.

## How variables are organized

Every variable belongs to a project stage. A stage is a named environment (like `development`, `staging`, or `production`) inside a project, which lives inside an org. Values are isolated per stage — there is no automatic promotion or inheritance between them.

```
org: acme-42
└── project: backend-api-1234
    ├── stage: development   ← DATABASE_URL = postgres://localhost/dev
    ├── stage: staging       ← DATABASE_URL = postgres://rds-staging/app
    └── stage: production    ← DATABASE_URL = postgres://rds-prod/app
```

Variables are encrypted at rest using a per-project data encryption key (DEK). The DEK itself is wrapped by a master key that lives outside the database. Plaintext is only produced during an authorized API request and is never persisted.

## Authentication

There are two ways to authenticate against the API:

- **Clerk JWT** — for web dashboard access and server-side integrations where your backend already issues Clerk tokens.
- **CLI access token** — issued by the device code flow (`barekey auth login`), valid for one hour and refreshable for up to 30 days.

Both tokens are passed as `Authorization: Bearer <token>` on every request.
